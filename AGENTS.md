## Контекст проекта

Я разрабатываю прототип формы для сайта `vashkontrol.ru`. Это сервис для оставления отзывов о российских государственных и муниципальных услугах, полученных в МФЦ и ведомствах.

Главная продуктовая проблема проекта: **люди не пишут отзывы**. Поэтому любые изменения должны уменьшать трение, ускорять выбор услуги/организации и делать форму понятнее, но не нарушать обязательные юридические и продуктовые ограничения.

Текущий репозиторий проекта: `Chychyndr/vashkontrol-review-form`.

Текущий формат проекта: статический сайт без сборщика. Основные файлы:

- `index.html`
- `styles.css`
- `script.js`
- `favicon.svg` или `assets/favicon.svg`
- `README.md`, если есть

## Главная цель интерфейса

Сделать форму отзыва максимально короткой, понятной и быстрой, чтобы пользователь мог:

1. Найти организацию или услугу.
2. Выбрать конкретную организацию и конкретную услугу.
3. Поставить оценки по обязательным параметрам.
4. При желании добавить комментарий, фото или видео.
5. Отправить отзыв без лишних шагов.

Основной сценарий должен быть таким:

> Пользователь знает, где получал услугу, поэтому сначала выбирает организацию, затем услугу. При этом должна оставаться возможность начать наоборот: сначала выбрать услугу, затем организацию.

## Жёсткие ограничения

Нельзя нарушать эти правила:

- Нельзя менять параметры оценки, то есть вопросы рейтинга.
- Нельзя менять шкалу оценки.
- Отзыв обязательно должен быть привязан к конкретной услуге и конкретной организации.
- Авторизация через Госуслуги/ЕСИА обязательна.
- Комментарий не должен становиться обязательным.
- Нельзя превращать отзыв в полноценное обращение по 59-ФЗ без отдельного сценария.
- Нельзя скрывать юридическое предупреждение про официальный ответ.
- Нельзя добавлять лишние обязательные шаги.
- Нельзя прятать кнопку отправки слишком далеко.
- У ведомств может не быть принадлежности к территории, поэтому нельзя делать территорию обязательной логикой для всех ведомств.
- Верхнюю шапку с параметрами оценки надо оставить: этап оценки и тип получения услуги, онлайн или очное получение.

## Параметры оценки, которые нельзя менять

В форме должны остаться именно эти вопросы:

1. `Время предоставления государственной услуги`
2. `Время ожидания в очереди при получении услуги`
3. `Вежливость и компетентность сотрудника, взаимодействующего с заявителем при предоставлении государственной услуги`
4. `Доступность информации о порядке предоставления государственной услуги`
5. `Комфортность условий в помещении, в котором предоставлена государственная услуга`

Шкала оценки — 5 звёзд. Все 5 оценок обязательны.

## Что уже решено по UX

### Поиск и выбор

- По умолчанию пользователь сначала выбирает **организацию**, потом **услугу**.
- Возможность начать с услуги должна остаться через вкладку `Услуги`.
- Вкладки должны идти в порядке: `Организации`, затем `Услуги`.
- Карточка организации или услуги должна выбираться нажатием по всей строке/карточке, а не отдельной кнопкой `Выбрать`.
- Не надо плодить отдельные блоки, кнопки и подсказки для городов. Подсказки должны быть компактно встроены в существующее поле поиска.
- Поле поиска должно быть понятным само по себе. Хороший placeholder: `Город, организация или услуга: Оренбург МФЦ, паспорт, ЕГРН`.
- Можно использовать `datalist` или похожие нативные подсказки в самом поле поиска.
- Не добавлять отдельный блок с быстрым выбором городов.
- Не показывать отдельную надпись с количеством найденных вариантов, если это увеличивает шум. Интерфейс должен оставаться компактным.

### Режимы поиска

В форме есть три режима:

- `Не знаю, помогите выбрать`
- `Получал в МФЦ`
- `Получал в ведомстве`

Логика фильтрации:

- Если выбран `Получал в МФЦ`, показывать только МФЦ и услуги, которые доступны через МФЦ.
- Если выбран `Получал в ведомстве`, показывать только ведомства и услуги, которые доступны в ведомствах.
- Если выбран `Не знаю, помогите выбрать`, показывать самые популярные варианты без жёсткого ограничения по типу организации.

### Популярность

- В списках сначала должны идти популярные организации/услуги.
- Не нужно показывать огромный список сразу на весь экран: сверху видны самые популярные варианты, остальное доступно прокруткой внутри списка.
- Блок `Часто ищут` должен быть коротким: около 5 пунктов, чтобы не занимать много места.

### Регион

- Регион может автоматически заполняться по доступной информации браузера.
- Пользователь всегда должен иметь возможность вручную изменить регион, даже если он был определён автоматически.
- Нельзя делать поле региона `readonly`.
- Над полем региона должна быть видимая подпись `Регион`, иначе непонятно, что это за поле.
- Не добавлять кнопку `Определить`. Автозаполнение должно происходить без действия пользователя.
- Прямое определение по IP невозможно только средствами фронтенда без бэкенда или внешнего сервиса. Во фронтенд-прототипе можно использовать только доступную браузерную информацию, например timezone.

### Дата

- Поле даты должно проверяться.
- Нельзя выбрать дату в будущем.
- Нельзя выбрать слишком далёкое прошлое. Текущий ориентир: не старше 3 лет.

### Видео

- В поле видео не упоминать YouTube.
- Использовать российские платформы, например: `RuTube, VK Видео, Дзен или другая российская платформа`.
- Видео остаётся необязательным.

### Комментарий

- Комментарий всегда необязательный.
- Не писать текст, который создаёт ощущение, что комментарий нужен обязательно.

### Официальный ответ

- Чекбокс `Хочу получить официальный ответ` должен остаться.
- Юридическое предупреждение/пояснение нельзя скрывать полностью.
- Текст не должен превращать отзыв в обращение по 59-ФЗ без отдельного сценария.
- Упоминания email/почтового адреса лучше не добавлять, если этого нет в сценарии.
- Допустимая логика: если пользователь выбирает официальный ответ, отзыв после модерации направляется в организацию для рассмотрения; статус и ответ можно посмотреть на сайте, уведомление может прийти на сайт или в MAX.

## Визуальный стиль

- Сайт должен выглядеть современно, чисто и официально, но не перегруженно.
- Основной цвет — зелёный, близкий к цвету сайта.
- Favicon должен быть простым зелёным кругом в цвет сайта.
- В шапке использовать простой логотип: щит + текст `Ваш Контроль`.
- Не вставлять неоригинальные сложные логотипы.
- Не добавлять лишние декоративные элементы, если они не помогают оставить отзыв быстрее.

## Технические правила

- Проект должен оставаться статическим сайтом, подходящим для GitHub Pages.
- Не добавлять сборщик, backend или зависимости без отдельной необходимости.
- Не ломать работу на GitHub Pages.
- Не использовать внешние API для определения региона в прототипе, если это не согласовано отдельно.
- Не хранить секреты в репозитории.
- Проверять, что `index.html`, `styles.css`, `script.js` связаны корректно.
- После изменения JS желательно проверять синтаксис командой:

```bash
node --check script.js
```

## GitHub Pages

Репозиторий может быть приватным, но GitHub Pages из приватного репозитория доступен не на всех тарифах GitHub. Для демонстрации проще использовать публичный репозиторий или платный тариф, если нужен приватный исходный код.

## Как отвечать и работать с изменениями

- Отвечать на русском.
- Делать изменения в коде, а не генерировать картинки вместо сайта.
- Если пользователь просит исправить сайт, нужно исправлять HTML/CSS/JS.
- Не добавлять новые обязательные шаги без явного запроса.
- Не усложнять интерфейс отдельными блоками, если можно решить задачу в существующем поле или компоненте.
- Если предлагается UX-решение, объяснять коротко и практически.
- Если изменение пушится в GitHub, лучше указывать commit SHA и кратко перечислять, что изменено.

## Главное правило продукта

Каждое изменение должно отвечать на вопрос:

> Помогает ли это пользователю быстрее и проще оставить корректно привязанный отзыв?

Если изменение добавляет шум, лишний выбор, обязательный шаг или юридическую неопределённость — его не делать.
